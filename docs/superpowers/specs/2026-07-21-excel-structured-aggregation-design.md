# Excel/CSV 结构化聚合设计

## 背景

当前知识库把 Excel、CSV、PDF、Word 等文件统一转成文本并按固定字符切片。用户询问平均值、求和、计数、最大值或最小值时，系统只能检索少量相关切片，再把这些切片交给大模型，因此无法保证对完整表格进行精确计算。

本设计是已批准的 `2026-07-15-offline-mixed-data-architecture-design.md` 的首个可交付结构化查询切片。它先解决 Excel/CSV 的精确聚合问题，再为后续 PDF、Word、PPT 表格抽取和混合问答保留接口。

## 目标

- 支持 `.xlsx` 和 `.csv` 文件的结构化解析。
- 默认把第一行作为表头；不同文件可以有不同字段。
- 自动推断字段类型和候选别名，并由管理员确认后发布。
- 支持 `avg`、`sum`、`count`、`min`、`max` 五种聚合。
- 支持基础筛选：分类字段等值筛选、数值范围和日期范围。
- 使用 ClickHouse 对完整已发布数据执行精确计算。
- 保持现有用户端对话 API 不变。
- 纯结构化问题优先返回确定性答案，不允许大模型改变计算结果。

## 非目标

- 本阶段不支持任意 SQL、自由表达式、跨数据集 JOIN、分组聚合或同比/环比。
- 本阶段不实现 PDF、Word、PPT 表格抽取。
- 本阶段不实现 `.xls`、`.xlsb`、`.xlsm` 或多行复杂表头。
- 本阶段不允许完全跳过管理员确认后直接发布自动推断结果。
- 本阶段不替换现有文档语义检索路径。
- 本阶段不一次性交付完整的离线混合数据平台、Qdrant 查询引擎或权限体系。

## 用户流程

1. 管理员上传 `.xlsx` 或 `.csv`。
2. 系统保留原文件，读取工作表和表头，并流式采样数据。
3. 数据源进入“待确认表结构”状态，而不是立即作为普通文本切片发布。
4. 管理端显示每个工作表的字段名、推断类型、样例值、空值率和候选别名。
5. 管理员确认或修正字段类型、查询别名和是否允许聚合/筛选。
6. 系统流式写入 Parquet staging，并批量导入 ClickHouse staging 表。
7. 行数、列数、类型转换和聚合校验通过后，原子更新 PostgreSQL 中的数据集发布指针。
8. 用户继续通过原有对话框提问；结构化路由识别聚合意图并查询已发布的 ClickHouse 表。

## 文件和工作表模型

- 一个上传文件仍对应一个知识来源 `source_id`。
- 一个 Excel 工作表对应一个结构化数据集；CSV 对应一个结构化数据集。
- 数据集使用稳定逻辑 ID，物理 ClickHouse 表使用带版本的不可变名称。
- 多工作表文件必须逐表确认；空工作表不发布。
- 第一行默认作为表头。空表头使用 `column_1` 等内部名称，但管理员必须提供可读名称后才能发布。
- 重复表头自动追加稳定后缀并要求管理员确认。
- 原始列名、规范化列名、查询别名和物理列名分别保存，不互相覆盖。

## 自动推断和确认

### 采样

- 使用 `openpyxl(read_only=True, data_only=True)` 流式读取 `.xlsx`。
- CSV 使用流式文本读取和编码探测。
- 每个工作表最多采样前 10,000 个非空数据行，不把整张表加载进内存。
- 确认后的正式导入仍逐行校验全部数据，采样结果不替代全量校验。

### 类型

第一阶段支持：

- `string`
- `integer`
- `decimal`
- `date`
- `datetime`
- `boolean`

推断冲突采用安全升级规则：`integer -> decimal -> string`。日期、布尔值和数值混杂时默认推断为 `string`，由管理员决定是否修正。管理员把字段确认成数值或日期后，全量导入遇到不可转换的非空值将使该文件版本进入失败状态，并报告行号、列名和脱敏样例，不静默丢弃。

公式单元格只读取缓存值，不在服务端重算。管理员确认用于聚合的公式列存在缺失缓存值时，发布失败并提示重新用办公软件计算并保存文件。

### 字段能力

管理员为每个字段确认：

- 展示名称和查询别名。
- 数据类型。
- 是否允许作为聚合指标。
- 是否允许作为筛选字段。
- 空值处理策略。

只有 `integer` 和 `decimal` 可以使用 `avg`、`sum`、`min`、`max`。`count` 可用于所有已发布字段或数据集总行数。

## 存储设计

### PostgreSQL

新增结构化控制面元数据：

- 数据集与工作表元数据。
- 字段定义、别名、类型和能力。
- schema 版本与校验状态。
- ingestion job、失败原因和发布指针。
- ClickHouse 物理表名、行数、内容哈希和发布时间。

PostgreSQL 不存储大规模表格数据，也不执行分析扫描。

### Parquet staging

- 正式导入以有界批次写入 Parquet。
- Parquet 文件位于配置的 `PARQUET_ROOT`，按 `source_id/dataset_id/version` 隔离。
- staging 完成前不覆盖已发布版本。
- 失败版本保留审计信息，临时数据按清理策略删除。

### ClickHouse

- 每个已确认工作表发布为版本化物理表。
- 表包含内部行号、source、dataset、version 和确认后的业务列。
- 表名和列名由服务端生成，不直接使用用户输入。
- 查询账户为只读账户；导入账户与查询账户分离。
- 查询必须设置执行时间、内存和结果行数限制。

## 查询路由

在现有文档 Agent 之前增加结构化查询路由：

1. 规则识别平均值、求和、计数、最大值、最小值等明确聚合意图。
2. 从已发布 schema 和别名中解析目标数据集、字段与筛选条件。
3. 如果唯一匹配，生成类型化 `StructuredQueryPlan`。
4. 如果字段或数据集存在歧义，返回确定性澄清信息，列出候选项；不退化为切片统计。
5. 如果问题不是结构化聚合问题，继续走现有文档检索和 Physoc 回答路径。

`deep` 模式不改变结构化计算结果。结构化查询始终扫描符合筛选条件的完整已发布数据，而不是最多五个切片。

## 安全查询计划

- 用户和大模型都不能提交原始 SQL。
- 查询计划只能引用已发布数据集、确认字段和允许的聚合函数。
- 使用 SQLGlot 构造并校验最终 SQL AST。
- 只允许单条 `SELECT`。
- 禁止子查询、文件函数、网络函数、系统表和任意表达式。
- 所有值通过参数绑定传入。
- 物理表名和列名只从已发布元数据解析。
- 默认 `max_execution_time=4` 秒、`max_result_rows=1000`，并设置内存限制。

## 回答行为

纯结构化聚合采用确定性模板返回，例如：

> 《销售数据.xlsx》的“订单金额”平均值为 12,345.67，基于 8,420 行有效数据计算，已排除 37 个空值。筛选条件：地区=华东。

回答同时保留：

- 来源文件和工作表。
- schema/数据版本。
- 指标、聚合函数和筛选条件。
- 总行数、有效值数量和空值数量。
- 查询耗时和审计 ID。

纯聚合默认不调用 Physoc，防止模型改写数值。后续混合问答可以把不可变结构化事实交给 Physoc 解释，但 Physoc 输出不得覆盖事实字段。

## 管理端改动

知识来源详情页增加“表结构”区域：

- 工作表列表和状态。
- 字段预览、样例值和推断类型。
- 类型、别名、聚合/筛选能力编辑。
- 确认并开始导入操作。
- 导入进度、发布版本和错误报告。

管理员确认前，用户端不能对该数据集执行结构化统计。旧的文本切片不会作为该表的权威统计来源。

## API 契约

新增管理端 API：

- `GET /api/knowledge/sources/{source_id}/structured-preview`
- `PUT /api/knowledge/sources/{source_id}/structured-schema`
- `POST /api/knowledge/sources/{source_id}/structured-publications`
- `GET /api/knowledge/sources/{source_id}/structured-status`

现有用户消息 API 保持不变。回答 bundle 可以增加可选结构化事实元数据，但不能破坏现有前端字段。

## 失败和降级

- 无表头、重复表头、全空文件：进入待管理员处理状态。
- 类型转换失败：停止发布新版本，继续服务旧版本。
- ClickHouse 不可用或超时：明确返回“结构化数据暂时不可用”，不使用切片猜测统计结果。
- 字段歧义：返回候选字段和工作表，不自动选择。
- 未确认 schema：提示管理员确认，不进入普通文本统计路径。
- CSV 编码无法可靠识别：拒绝发布并显示支持的编码建议。
- 新版本行数或基础聚合校验失败：保留旧发布指针。

## 部署和配置

复用现有 ClickHouse、PostgreSQL 和 `PARQUET_ROOT` 配置。新增独立的 ClickHouse 导入与只读查询凭据，以及结构化查询开关。默认保持旧路径，目标环境完成 migration、ClickHouse 集成测试和代表性数据验证后再开启。

建议环境变量：

- `STRUCTURED_QUERY_ENABLED=false`
- `CLICKHOUSE_INGEST_USER`
- `CLICKHOUSE_INGEST_PASSWORD_FILE`
- `CLICKHOUSE_QUERY_USER`
- `CLICKHOUSE_QUERY_PASSWORD_FILE`
- `STRUCTURED_QUERY_TIMEOUT_SECONDS=4`
- `STRUCTURED_INGEST_BATCH_ROWS=50000`

## 测试和验收

### 单元测试

- XLSX/CSV 表头、类型、别名和多工作表推断。
- 混合类型、安全升级、空值和公式缓存处理。
- schema 确认、版本和状态迁移。
- 聚合意图、字段匹配、歧义和筛选解析。
- SQLGlot 白名单和危险 SQL 拒绝。
- 确定性回答格式和数值不可变性。

### 集成测试

- Parquet staging 与 ClickHouse 批量导入。
- `avg/sum/count/min/max` 与 Python 基准计算一致。
- 数值、日期和分类筛选结果一致。
- 发布失败时旧版本继续可用。
- ClickHouse 超时和不可用时不退化为切片统计。
- 现有文档问答和 Physoc 路径不回归。

### 首批验收用例

- 单工作表平均值。
- 多工作表同名字段歧义。
- 带分类筛选的平均值。
- 带日期范围的求和。
- 空值列的有效值数量。
- 混合类型列发布失败。
- 一百万行合成数据流式导入，内存不随行数线性增长。
- 15 个并发用户执行常用聚合时，目标 p95 不超过 5 秒。

## 分阶段交付

1. schema 推断和管理端确认。
2. Parquet staging、ClickHouse 导入和版本发布。
3. 聚合意图、受限查询计划和确定性回答。
4. 对话 API 接入、部署开关和回归验证。
5. 后续扩展 PDF、Word、PPT 表格抽取及混合问答。
