# 通用质量评测工作台设计方案

## 目标

将现有的单条检索评测页面扩展为业务无关、可重复使用的质量评测工作台。即使当前没有真实公司业务数据，系统也能先完成能力建设；以后获得真实资料和问题集时，可以直接导入使用，无需重新设计。

正式数据库默认保持空白。示例数据只允许存在于自动化测试夹具中，不得写入正在运行的正式或本地业务环境。

## 实施范围

本阶段包含：

- 从 XLSX、CSV 和 JSON 文件批量导入评测问题。
- 写入数据库前提供数据预览和逐行校验。
- 支持管理员自定义分类、标签和外部唯一标识。
- 将选中的问题或筛选结果作为一个命名批次在后台运行。
- 记录每个评测批次使用的检索阈值。
- 生成汇总指标、失败案例列表和批次对比结果。
- 保留现有的单条创建、运行、删除和检索诊断能力。

本阶段不包含：

- 预设部门、业务领域或公司专属规则。
- 向正式数据库写入演示评测数据。
- 根据公司文档自动生成标准答案。
- 向用户侧应用暴露资料片段、检索诊断或评测控制功能。
- 自动把评测阈值应用到正式问答链路。正式问答继续使用 `RETRIEVAL_MIN_SCORE`，以后需要调整时由管理员通过部署配置明确修改。
- 身份认证和路由权限管理。

## 产品结构

质量评测继续作为管理员专用模块，并拆分成页面级工具：

- `/quality/cases`：管理评测集、筛选、手工创建、批量导入、删除和发起批次运行。
- `/quality/reports`：查看评测批次历史和汇总对比。
- `/quality/reports/:batchId`：查看单个批次的报告、失败案例和检索诊断。

现有 `/quality` 路由重定向到 `/quality/cases`。

页面不增加装饰性英文小标题或功能说明文案。管理端常规文字不得低于 12px，所有展示时间统一使用 `YYYY-MM-DD hh:mm:ss`。

## 组件划分

- `QualityCasesPage.vue`：评测集页面的路由级组合和查询状态管理。
- `QualityReportsPage.vue`：评测批次历史与对比页面的路由级组合。
- `QualityReportDetailPage.vue`：单个评测报告页面的路由级组合。
- `EvaluationCaseToolbar.vue`：筛选、批量选择、导入和发起评测。
- `EvaluationCaseList.vue`：现有评测问题列表，增加选择、分类和标签展示。
- `EvaluationCaseFormDialog.vue`：保留现有的手工创建评测问题能力。
- `EvaluationImportDialog.vue`：选择文件、预览数据、展示校验错误并确认导入。
- `EvaluationBatchDialog.vue`：设置批次名称和仅用于评测的检索阈值。
- `EvaluationBatchList.vue`：展示历史批次、状态和执行进度。
- `EvaluationReportSummary.vue`：展示汇总指标和批次差异。
- `EvaluationFailureList.vue`：展示未通过和发生误召回的案例。
- `EvaluationRunDetail.vue`：复用现有的单条检索诊断组件。
- `useQualityCases.ts`：负责问题加载、筛选、选择、导入和增删操作。
- `useEvaluationBatches.ts`：负责批次启动、轮询、报告和批次对比。

路由级组件只负责页面组合。接口状态和副作用分别放入两个功能组合式函数中；子组件使用明确的 TypeScript Props 和 Emits 契约通信。

## 导入数据契约

### 支持的文件

- XLSX：只读取第一个工作表。
- CSV：支持 UTF-8 和 UTF-8 BOM。
- JSON：文件内容必须是评测问题对象数组。

单个文件最大 5MB，最多包含 2,000 行数据。空文件或不存在有效数据的文件直接拒绝导入。

### 字段定义

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `question` | 是 | 问题不能为空，最大 1,000 个字符。 |
| `expect_answer` | 是 | 支持 `true/false`、`1/0` 和中文是否标签。 |
| `expected_sources` | 否 | 期望资料名称，多个值使用 `|` 分隔，并与已上传资料进行匹配。 |
| `expected_terms` | 条件必填 | 期望命中的关键词，多个值使用 `|` 分隔。 |
| `category` | 否 | 管理员自定义分类，最大 80 个字符。 |
| `tags` | 否 | 管理员自定义标签，多个值使用 `|` 分隔，系统自动去重。 |
| `top_k` | 否 | 默认值为 5，允许范围为 1 至 10。 |
| `external_key` | 否 | 调用方提供的稳定唯一标识，最大 120 个字符。 |

当 `expect_answer=true` 时，至少需要设置一项可匹配的期望资料或期望关键词。当 `expect_answer=false` 时，不要求设置期望资料和关键词。

### 两阶段导入流程

1. `POST /api/admin/evaluations/import/preview` 接收文件上传。
2. 后端解析并标准化数据，匹配资料名称，返回有效行、逐行错误、重复警告和数量统计，此时不写入评测问题表。
3. 管理员在页面中检查预览结果。
4. `POST /api/admin/evaluations/import/confirm` 接收短期有效的预览令牌，并将校验通过的数据写入数据库。

重复判断优先使用 `external_key`；没有该字段时，使用标准化后的问题文本。第一版本只跳过并报告重复数据，不覆盖已有评测问题。

预览令牌由服务端保存，有效期为 30 分钟，避免前端在预览后静默修改已校验的数据。

## 数据模型

### `evaluation_cases` 新增字段

- `category`：可为空的分类字符串。
- `tags`：JSON 字符串数组，默认为空数组。
- `external_key`：可为空的外部标识，并建立索引。
- `import_batch_id`：可为空的导入批次标识，用于追溯数据来源。

### `evaluation_import_batches`

包含以下字段：

- `id`
- `file_name`
- `status`
- `total_rows`
- `valid_rows`
- `invalid_rows`
- `duplicate_rows`
- `created_at`
- `completed_at`

### `evaluation_batches`

包含以下字段：

- `id`
- `name`
- `status`
- `retrieval_min_score`
- `case_count`
- `completed_count`
- `passed_count`
- `failed_count`
- `false_positive_count`
- `started_at`
- `completed_at`
- 可为空的 `error_message`

### `evaluation_runs` 新增字段

- `batch_id`：可为空的评测批次外键。现有手工运行的单条评测记录保持为空，不影响原有能力。

数据库初始化逻辑需要为已有 PostgreSQL 环境补充兼容字段和新表，不删除现有评测问题和运行记录。

## 批次执行流程

管理员可以运行勾选的问题，也可以运行当前筛选条件下的全部问题。创建批次时必须填写批次名称，并可以设置一个可选的最低检索分数；没有填写时记录后端当前默认阈值。

`POST /api/admin/evaluations/batches` 创建状态为 `queued` 的批次后立即返回。FastAPI 使用后台任务逐条执行，并在每条问题完成后更新批次进度。前端在批次状态为 `queued` 或 `running` 时定时轮询批次接口。

批次设置的阈值只传递给评测检索逻辑，不修改进程环境变量，也不影响 DCAgent 用户侧的正式问答链路。

单条问题执行异常时，该问题生成失败诊断，批次继续运行。只有批次无法启动或批次级持久化失败时，整个批次才标记为 `failed`。

## 报告指标

每个完成的批次需要提供：

- 问题总数、通过数和未通过数。
- 整体通过率。
- 应有答案问题通过率。
- 无答案识别准确率。
- 误召回数量和误召回率。
- 设置了期望资料的问题的平均资料召回率。
- 设置了期望关键词的问题的平均关键词召回率。
- 平均最高分和批次最高分。
- 按分类和标签汇总的问题数量及通过情况。

失败案例列表支持按照资料缺失、关键词缺失、无命中和误召回进行筛选。点击失败案例后，使用现有诊断组件展示关键词分数、向量分数和综合分数。

报告页面支持选择两个已完成批次进行比较，展示指标变化以及状态发生变化的问题。两个批次的问题集合不完全相同时，系统分别展示共享问题的变化和完整批次的整体指标，并明确标注统计口径。

## 接口设计

新增接口：

- `POST /api/admin/evaluations/import/preview`
- `POST /api/admin/evaluations/import/confirm`
- `GET /api/admin/evaluations/cases`，支持分类、标签、答案预期和运行状态筛选。
- `POST /api/admin/evaluations/batches`
- `GET /api/admin/evaluations/batches`
- `GET /api/admin/evaluations/batches/{batch_id}`
- `GET /api/admin/evaluations/batches/compare?left=...&right=...`

本阶段继续兼容现有评测接口，确保当前管理端功能和自动化测试不被破坏。

## 异常处理

- 导入错误需要明确指出行号、字段名称和可读的中文原因。
- 资料名称不存在或存在多个同名资料时，作为当前行的校验错误处理，不允许静默忽略。
- 不支持的文件类型、超出大小限制的文件和已过期的预览令牌返回明确的 4xx 响应。
- 批次轮询失败时保留最后一次已知进度，并提供重新加载操作。
- 批次级持久化失败时保存管理员可读的错误信息，不向页面暴露堆栈信息。
- 确认导入使用数据库事务：所有校验通过的数据全部写入，或者全部不写入。

## 空状态

没有真实业务数据时，评测集页面保持空状态，只提供“手工创建”和“导入文件”两个操作，不自动创建示例问题。

导入弹窗可以下载空白模板。模板只包含字段表头和填写说明，不包含任何模拟公司问题。

## 测试方案

后端测试覆盖：

- XLSX、CSV 和 JSON 解析。
- 文件编码、行数限制、数据标准化和可读校验错误。
- 资料名称匹配，以及同名或缺失资料的错误处理。
- 重复数据跳过和事务化确认导入。
- 数据库兼容迁移。
- 后台批次进度，以及单条失败后继续执行。
- 指标汇总和批次对比。
- 评测专用阈值不会改变正式检索配置。
- 空数据库不会自动写入演示数据。

管理端测试覆盖：

- 导入预览、错误行、重复警告和确认导入。
- 分类与标签筛选，以及问题批量选择。
- 发起批次和进度轮询。
- 空状态、加载状态、失败状态、完成状态和批次对比状态。
- 页面级路由拆分和移动端溢出检查。
- 所有常规可见文字不低于 12px。

最终验证包括后端单元测试、管理端 Vitest 测试、管理端生产构建，以及桌面端和移动端浏览器检查。

3. 完成批次执行、指标汇总、批次对比接口及对应测试。
4. 拆分管理端路由，实现导入流程、问题筛选和批量选择。
5. 实现批次报告与对比页面，完成响应式检查和全量回归验证。
